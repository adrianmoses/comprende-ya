---

## Why

Phrase Autopsy is one of the three editorial pillars of the product (alongside MCQs and the Chunk Library). 016 ships the panel UI against a hand-written frontend fixture covering two phrases on one seeded video — useful for proving the visual + interaction work, but it does not give Ana real data on real videos. 017 is the backend that lets *any* phrase Ana taps return a Spanish-internal grammar breakdown plus a "why it sounds native" note, generated by Claude, persisted so the next tap (and the next session, and any future caller) is free.

Phrase Autopsy is the moment Ana stops, points at something that *sounded* native but she couldn't parse, and gets a Spanish-only explanation of why it works that way. Without 017, the panel only ever has the 2–3 phrases the fixture happens to include for the seeded video; the feature is permanently a demo. With 017, every phrase Ana cares about can be inspected.

### Consumer Impact

- **Direct consumer: the Escuchando frontend (`webapp/src/routes/listen.$id.tsx`) via 018.** Once 018 ships tappable transcript words, every tap turns into a `POST /api/videos/{id}/autopsy/explain` call and the existing `AutopsyPanel` (016) renders the response. The panel internals don't move — only its data source.
- **Indirect consumer: Ana, the single B2 learner.** She can interrogate the language instead of being quizzed on it. Tap → Spanish-internal explanation → close → keep listening. The audio stays paused at her place; her saved phrases (when 020 lands) carry the autopsy with them.
- **020 (Mis frases / Chunk Library) re-uses the same persisted entries.** When Ana saves a phrase from the autopsy panel, the chunk library can hold a reference to the autopsy row rather than copying the data.

### Roadmap Fit

- **Depends on:** nothing in the backend. The Claude integration pattern is already established in `services/questions.py` and `services/dialect_classifier.py`.
- **Soft-couples to:** 016 (the existing panel UI is the eventual renderer) and 018 (tappable transcript words are the natural trigger). 017 ships the endpoint; the frontend integration is a separate small task that lands with 018, at which point 016's fixture is retired.
- **Unblocks:** 018 (which needs an autopsy endpoint to call) and 020 (which can persist saved phrases referencing autopsy rows).
- **Doesn't block:** any frontend work that's already shipped — 016 stays fixture-backed until 018 lands tappable words.

## What

### Acceptance Criteria

- [ ] **`POST /api/videos/{video_id}/autopsy/explain`** accepts `{ phrase: string, start_time: number }` and returns an `AutopsyEntry` with `{ id, video_id, phrase, start_time, register, grammar: [{tag, text}], natural_notes: [string], created_at }`. `video_id` here is the YouTube id, matching the rest of `/api/videos/{video_id}/...`.
- [ ] **Cache hit returns immediately.** If a row already exists for `(video.id, normalized_phrase)`, the endpoint returns it without calling Claude. Cache hit and cache miss return the same response shape.
- [ ] **Cache miss generates via Claude** using a Spanish-only prompt, parses the JSON response, persists the row, and returns it. Total round-trip on cache miss is bounded by Claude latency (no other I/O of consequence).
- [ ] **`GET /api/videos/{video_id}/autopsy`** returns all cached `AutopsyEntry` rows for the video (Spanish-only, no English fields). May be empty. Consumed by 020 to surface a "saved phrases" list, and by 016's eventual fixture-replacement once 018 is in.
- [ ] **Phrase normalization for the cache key.** Trim whitespace; collapse internal whitespace to single spaces; lowercase for the lookup key only (the stored `phrase` keeps original casing for display). `«No me da igual»` and `no me da igual` hit the same cache row.
- [ ] **Data shape is Spanish-only.** No `natural` (English translation) or `literal` field. Permanent per OVERVIEW §Non-Goals.
- [ ] **Validation.** 422 if `phrase` is empty / >200 chars, or `start_time` is negative. 404 if the video doesn't exist.
- [ ] **Claude failure handling.** If Claude returns malformed JSON or the call errors, the endpoint returns 502 with a Spanish error message; nothing is persisted.
- [ ] **Alembic migration** creates the `phrase_autopsy` table with appropriate indexes and a unique constraint on `(video_id, phrase_key)` where `phrase_key` is the normalized lookup key.
- [ ] **Tests in the existing pytest suite** cover: cache hit, cache miss with Claude mocked, malformed Claude response, video-not-found, validation errors, normalization (two casings hit one row).
- [ ] **Spike artefact.** Before locking the prompt, ~5–10 real phrases drawn from a seeded video are sent through a draft prompt; the output is captured (in the decision record, not the codebase) and inspected for JSON parseability, grammar-tag consistency, and B2-appropriate `natural_notes`. The spec moves from `draft` to `approved` only after this passes.
- [ ] `ruff check` and `ruff format` clean.

### Non-Goals

- **Pre-generating autopsies during `process_video_flow`.** Decided against — it would generate entries Ana may never tap and adds an N×Claude-call cost to every ingest. On-demand, cached, fills over time.
- **Replacing 016's frontend fixture.** 016 stays on the fixture until 018 lands tappable words. With on-demand-only, the "Frases destacadas" list trigger has nothing to populate from until users start tapping; 016's fixture remains the demo affordance.
- **English `natural` / `literal` fields.** Permanent product position (OVERVIEW §Non-Goals), not a deferral.
- **Quality / confidence score on entries.** The data shape doesn't carry it; if it ever matters, it's a follow-up.
- **Regeneration / edit / delete endpoints.** Cache is write-once. If a row is bad, fix the prompt and clear the row manually. No `PATCH /autopsy/{id}` surface.
- **Cache TTL or invalidation.** Autopsy of a given phrase doesn't drift over time. Migrating to a new prompt or new model can be handled by an Alembic data migration if it ever matters.
- **Token-level transcript / tappable words.** That's 018. 017 only assumes the *caller* knows the phrase string and an approximate `start_time`.
- **Autopsy across videos.** The cache is per-video; the same phrase in two different videos gets two rows. Phrases like *"no me da igual"* aren't unique across videos, and the surrounding context (which the Claude call uses) is video-specific.
- **Multi-user / per-user cache.** Single-user product (OVERVIEW §Non-Goals); cache is global per video.

### Open Questions

1. **Does the prompt need surrounding-segment context to generate good autopsies?** Likely yes — "no me da igual" by itself is parseable, but a phrase that depends on prior discourse (anaphora, irony, register cues) won't be. Recommendation: the endpoint resolves the segment containing `start_time` (via `SegmentsRepository`) and passes the immediately surrounding 1–2 segments as context to Claude. The spike should confirm this is necessary.
2. **Phrase length cap of 200 chars.** Plenty for a 1–2-sentence colloquial phrase; rejects accidental "paste a paragraph." Adjustable post-spike if real usage shows different.
3. **Should the YouTube-id lookup live at the route layer or inside the service?** Pattern in the codebase is route-layer resolution (`videos.py` resolves YouTube id → `Video` row, then passes the ORM object down). Following that pattern keeps the service unaware of the routing convention.

#### Database model

```python
# src/models/database.py
class PhraseAutopsy(SQLModel, table=True):
    __tablename__ = "phrase_autopsy"

    id: Optional[int] = Field(default=None, primary_key=True)
    video_id: int = Field(foreign_key="videos.id", index=True)
    phrase: str                          # original casing for display
    phrase_key: str = Field(index=True)  # normalized for cache lookup
    start_time: float
    register: str
    grammar: str                         # JSON string of [{tag, text}]
    natural_notes: str                   # JSON string of [string]
    created_at: datetime = Field(default_factory=datetime.utcnow)

    video: Optional[Video] = Relationship(back_populates="phrase_autopsies")
```

Add a `phrase_autopsies` back-reference on `Video`. Unique constraint on `(video_id, phrase_key)` declared via `__table_args__`.

#### Pydantic schemas

```python
# src/models/schemas.py
class AutopsyGrammarRow(BaseModel):
    tag: str
    text: str

class AutopsyExplainRequest(BaseModel):
    phrase: str = Field(min_length=1, max_length=200)
    start_time: float = Field(ge=0)

class AutopsyEntryResponse(BaseModel):
    id: int
    video_id: str               # YouTube id (for symmetry with the rest of /api/videos)
    phrase: str
    start_time: float
    register: str
    grammar: list[AutopsyGrammarRow]
    natural_notes: list[str]
    created_at: datetime
```

#### Service

`src/services/phrase_autopsy.py` follows the singleton pattern used by `questions.py`. One public method:

```python
class PhraseAutopsyService:
    def __init__(self):
        self._client = anthropic.Anthropic(api_key=settings.ANTHROPIC_API_KEY)

    def explain(
        self,
        phrase: str,
        context_segments: list[str],   # 1-2 surrounding transcript segments
    ) -> AutopsyPayload:
        """Returns {register, grammar: [{tag, text}], natural_notes: [str]}.

        Raises AutopsyGenerationError on parse failure or Claude error.
        """
```

The prompt is Spanish-only and asks Claude to:

1. Identify the register (`cotidiano · neutral`, `formal · escrito`, etc.).
2. Produce 2–5 `{tag, text}` grammar rows where `tag` is short (≤25 chars, e.g. *"preposición"*, *"subjuntivo de duda"*) and `text` is one Spanish sentence explaining its role in this phrase.
3. Produce 2–4 `natural_notes` — short Spanish observations on why it sounds native, register cues, common alternatives, when not to use it.
4. Return strict JSON matching the response schema. No prose outside the JSON.

The model is the same one already used by `questions.py` (`claude-4-sonnet-20250514`) for now — refresh to current models is its own roadmap item (026), out of scope here.

The service does not know about caching, the database, or YouTube ids. It takes a phrase + context and returns a payload. The route layer handles cache lookup and persistence.

#### Repository

```python
# src/repositories/autopsy_repository.py
class AutopsyRepository:
    def __init__(self, session: Session): ...

    def get_by_phrase(self, video_id: int, phrase_key: str) -> PhraseAutopsy | None: ...
    def list_for_video(self, video_id: int) -> list[PhraseAutopsy]: ...
    def create(self, video_id: int, phrase: str, start_time: float, payload: AutopsyPayload) -> PhraseAutopsy: ...
    def to_response(self, row: PhraseAutopsy, youtube_id: str) -> AutopsyEntryResponse: ...
```

`phrase_key` is computed by a module-level `normalize_phrase(phrase: str) -> str` helper (trim, collapse whitespace, casefold). Same helper used by both `get_by_phrase` and `create` so they can never disagree.

#### Routes

```python
# src/api/routes/videos.py
@router.post("/{video_id}/autopsy/explain", response_model=AutopsyEntryResponse)
def explain_phrase(video_id: str, body: AutopsyExplainRequest, session: Session = Depends(get_session)):
    video = video_repo(session).get_by_youtube_id(video_id)
    if not video:
        raise HTTPException(404, "Video no encontrado")

    repo = AutopsyRepository(session)
    key = normalize_phrase(body.phrase)
    cached = repo.get_by_phrase(video.id, key)
    if cached:
        return repo.to_response(cached, video.youtube_id)

    context = SegmentsRepository(session).context_around(video.id, body.start_time, window=1)
    try:
        payload = phrase_autopsy_service.explain(body.phrase, context)
    except AutopsyGenerationError as exc:
        raise HTTPException(502, f"Generación fallida: {exc}") from exc

    row = repo.create(video.id, body.phrase, body.start_time, payload)
    return repo.to_response(row, video.youtube_id)


@router.get("/{video_id}/autopsy", response_model=list[AutopsyEntryResponse])
def list_autopsies(video_id: str, session: Session = Depends(get_session)):
    video = video_repo(session).get_by_youtube_id(video_id)
    if not video:
        raise HTTPException(404, "Video no encontrado")
    rows = AutopsyRepository(session).list_for_video(video.id)
    return [AutopsyRepository(session).to_response(r, video.youtube_id) for r in rows]
```

`SegmentsRepository.context_around(video_id, t, window)` is a tiny new method that returns the segments overlapping `[t-window_segments, t+window_segments]`. If the segments table is empty for this video, return `[full_transcript]` as a fallback so the endpoint still works on legacy rows.

#### Migration

Single Alembic revision adding `phrase_autopsy` with the columns above, the foreign key on `video_id`, an index on `video_id`, an index on `phrase_key`, and a unique constraint on `(video_id, phrase_key)`. Generated via `uv run alembic revision --autogenerate -m "add phrase_autopsy table"`, manually checked.

#### Spike

Before merging this spec or writing any of the above code:

1. Draft the prompt locally in a one-off script.
2. Pick 5–10 phrases from a seeded video's transcript — mix idiomatic, grammatical, and ordinary; include at least one that's only meaningful in context.
3. Run each through the prompt against the current model. Inspect:
   - JSON parses without `json.loads` errors.
   - Grammar tags are short, distinct, in Spanish, and don't repeat across rows.
   - `natural_notes` are 2–4 entries, B2-readable, no English crutches, no "this sounds natural because it sounds natural" tautologies.
   - The prompt produces consistent output across re-runs (sample twice with different seeds; check stability).
4. Capture the prompt, the inputs, and a representative output in the decision record (`017-phrase-autopsy-data-model/decision.md`) once the feature lands. Do not commit a transcript of every spike run — one good and one bad sample is enough.

If the spike fails (Claude can't reliably produce the shape), revisit the schema before the rest of the implementation. Possible adjustments: drop `register` to a free-form string-or-null, or relax `grammar` row count.

### Confidence

**Level:** Low (until the spike), Medium thereafter.

**Rationale:**

The plumbing — endpoint + service + repository + Alembic migration + tests — is well-understood and analogous to the existing question-generation path (`services/questions.py`, `repositories/...`, `api/routes/videos.py`). The risk lives entirely in one question: *can Claude reliably produce the structured Spanish-only autopsy shape that 016 expects?* The 016 fixture was hand-written — there's no existing evidence that Claude generates the same quality consistently. A 30-minute spike resolves this before the rest of the work commits to a schema.

Once the spike confirms the shape, confidence rises to Medium. The reason it's not High even then: the data quality bar is editorial — "is this autopsy actually useful to a B2 learner?" is a softer question than "does it parse and persist?" Iterating on prompt quality is expected, but the storage shape should be stable after the spike.

**Validate before proceeding:**

- Run the spike (5–10 phrases through a draft prompt; inspect output).
- Confirm the surrounding-segment context window (1–2 segments) is enough — if context-dependent phrases produce empty grammar rows or generic notes, expand the window.
- Confirm 200-char phrase cap is reasonable against real candidate phrases from seeded videos.

### Key Decisions

- **On-demand per tap, cached — not pre-generated.** Decided in discovery. Pre-generation would generate entries Ana may never tap and ties Phrase Autopsy quality to a candidate-selection heuristic that nobody has designed. On-demand fills the cache only with phrases that real usage produced. Cost: 016 stays on its fixture until 018 ships tappable words.
- **Cache is write-once, no TTL.** Phrase autopsy doesn't drift. If the prompt is bad, fix the prompt and bump a migration to clear the cache.
- **Cache key is `(video_id, phrase_key)` not `(phrase_key)` alone.** Same phrase, different videos = different surrounding context = different autopsy. Matches the eventual API shape (`/api/videos/{id}/autopsy`).
- **Phrase normalization is whitespace + casefold only.** No accent stripping, no punctuation stripping. Matches a "tap a literal range of the transcript" mental model — Ana taps `«no me da igual»` and gets back the autopsy of *that*, not a punctuation-insensitive normalized form.
- **Service does not know about caching.** Cleaner test boundaries — service tests mock Claude only; route tests cover cache logic with the service mocked.
- **Re-use the existing Anthropic singleton pattern + model id.** The model refresh (Opus 4.7 / Sonnet 4.6 / Haiku 4.5) is roadmap item 026, scoped separately. 017 doesn't widen its blast radius by also doing model selection.
- **YouTube id at the route layer, DB id below.** Matches every existing route in `videos.py`. The `AutopsyEntryResponse.video_id` field returns the YouTube id for caller symmetry.

### Testing Approach

Follows the test infra established in 025 (pytest + DB fixture + monkeypatched service singletons).

**Service-level tests** (`tests/services/test_phrase_autopsy.py`):

- Claude returns valid JSON → service returns parsed `AutopsyPayload`.
- Claude returns malformed JSON → `AutopsyGenerationError`.
- Claude returns JSON missing required fields → `AutopsyGenerationError`.
- Prompt construction includes the phrase and context segments verbatim.

**Route-level tests** (`tests/api/test_autopsy_routes.py`):

- `POST /autopsy/explain` cache miss → calls service → persists → returns entry.
- `POST /autopsy/explain` cache hit → does NOT call service → returns cached row.
- `POST /autopsy/explain` two casings of the same phrase → second call is a cache hit.
- `POST /autopsy/explain` empty phrase → 422.
- `POST /autopsy/explain` phrase >200 chars → 422.
- `POST /autopsy/explain` negative `start_time` → 422.
- `POST /autopsy/explain` unknown YouTube id → 404.
- `POST /autopsy/explain` service raises `AutopsyGenerationError` → 502, no row persisted.
- `GET /autopsy` returns all cached rows for a video, empty list when none.
- `GET /autopsy` unknown YouTube id → 404.

**Migration smoke** (covered implicitly by the existing `tests/conftest.py` fixture which runs migrations on the SQLite test DB): the new revision applies and the unique constraint blocks duplicates.

**No frontend tests.** Same deferral as 013–016. The frontend integration with 017 ships with 018, not here.

**Out of test scope:** real Claude calls. The spike captures the qualitative bar in the decision record; CI tests use mocks.
