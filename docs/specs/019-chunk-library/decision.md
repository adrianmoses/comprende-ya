## Context

019 ships the persistence layer for Mis frases — the saved-phrase library that 020 will render against. The product already had a session-local "guardar" toggle in `AutopsyPanel` (016), but it forgot saves on refresh; 019 makes them durable, attaches Claude-generated speaking prompts at save time, and exposes a global list / delete surface.

The implementation followed the spec's "near-clone of 017" framing very closely. The only meaningful discoveries during build were:

1. The Claude prompt validated the spike with no further iteration — 14/14 mechanical pass, 28/28 qualitative pass on the first prompt draft (see `Test Evidence`).
2. The autogenerated Alembic migration produced false-positive churn on three unrelated tables (`answer_progress`, `frase_exercise`, `processing_jobs`) due to SQLite + SQLAlchemy type-coercion noise. The diff had to be hand-stripped to keep only the new-table ops.
3. One service test (`test_generate_object_not_array_raises`) was originally written assuming the JSON extractor was strict; in fact `_extract_json` (lifted from 017) is *lenient* — it salvages an array nested inside an object wrapper. The test was rewritten to use a payload with no array substring at all, and the lenient behavior is now documented inline.
4. A `/simplify` pass after the green build collapsed `ChunkRepository.to_response` from a 3-arg signature to a 1-arg signature and added `selectinload(Chunk.video)` to `list_all()` to kill the latent N+1 on `GET /api/chunks`. The fixture `seeded_video`, which was duplicated byte-for-byte across `tests/test_chunks_routes.py` and `tests/test_autopsy_routes.py`, was hoisted to `tests/conftest.py`.

The two Open Questions the spec parked (length cap 200 vs ~80; normalization aggressiveness) remain unresolved — both rely on real production signal that doesn't exist yet.


## Decision

Built the chunk library as a thin standalone table (`chunks`) with a single new top-level router at `/api/chunks` (POST 201 / GET / DELETE 204). Saves trigger one Claude call to generate 2-4 Spanish speaking prompts, which persist on the row as a JSON string. Cache key is `(video_id, phrase_key)` reusing 017's `normalize_phrase` helper — same-phrase saves on the same video collapse to one row regardless of casing or interior whitespace.

The chunk row does NOT FK to `phrase_autopsy`. If Mis frases ever wants to enrich the row with grammar/register data, it resolves `(video_id, phrase_key)` against the autopsy cache at read time. No schema coupling.

Service / repository / route split mirrors 017 exactly: service knows nothing about caching or DB, repository owns the (video_id, phrase_key) lookup and JSON (de)serialization, route layer composes them. 502 on Claude failure, 422 on Pydantic validation, 404 on unknown video, 201 on save (including cache hit), 204 on delete.

## Alternatives Considered

### Chunk → PhraseAutopsy relationship

**Option A:** FK to `phrase_autopsy` — every chunk references an autopsy row; chunks carry only practice metadata; grammar/register/natural_notes live on the autopsy.
- Pros: no data duplication; "save with autopsy" is the natural flow in 016/017; storage-efficient.
- Cons: a chunk can't be saved for a phrase that doesn't have an autopsy yet (e.g. a hand-typed save from a future search-bar feature); couples 019's mental model to 017's existence.

**Option B (chosen):** Standalone chunks — chunk row is self-contained for everything Mis frases renders; no FK to autopsy.
- Pros: chunks don't require an autopsy to exist; the schema doesn't enforce the canonical-origin assumption that may not hold long-term; Mis frases can still resolve autopsy data by `(video_id, phrase_key)` lookup if it wants to.
- Cons: linguistic enrichment costs a second lookup; "what data does a chunk carry" lives in two places (the row + the optional autopsy row).

**Chosen:** B. Decided in discovery — the design's mental model is "save = bookmark," not "save = bookmark + explanation." Standalone keeps 019's contract honest and decouples cleanup if 017's data shape ever changes.

### Where do the practice prompts come from?

**Option A:** Hand-curated or fixture, generated later. Smallest 019 scope.
- Pros: zero Claude cost in 019; defers the "is the prompt any good?" question.
- Cons: Mis frases ships empty-of-prompts until a future feature; defeats the whole point of the screen.

**Option B (chosen):** Claude-generated at save time, persisted on the row. One call per `POST /chunks`.
- Pros: matches 017's on-demand-and-cache pattern; Ana sees *her* prompts on every render, not a re-rolled set; cache is write-once so a bad row is deleted and re-saved.
- Cons: extra Claude cost per save (one call, bounded latency).

**Option C:** Claude-generated on demand, not stored. `GET /chunks/{id}/prompts` re-rolls each viewing.
- Pros: no staleness; cheaper upfront.
- Cons: breaks the "these are *my* prompts" mental model — every render gives Ana different prompts.

**Chosen:** B. Discovery question; B was preferred over A *and* C.

### Should `mastery` ship in 019?

**Option A:** Include as manual slider — Ana sets her own 0-1 value via PATCH.
- Pros: matches the design fixture's numeric mastery values; useful even without recordings.
- Cons: ships a column whose semantics depend on user discipline; no objective signal.

**Option B:** Include with default 0, no update path — column exists, value is always 0 until 021.
- Pros: schema-stable through 021's recordings feature.
- Cons: half-empty column from day one; smell.

**Option C (chosen):** Defer entirely — no mastery column in 019.
- Pros: smallest 019 surface; column lands with 021 when the recording signal that drives it exists.
- Cons: Mis frases UI has no mastery to render against in 020 — the design's mastery dot becomes a 020 mock until 021.

**Chosen:** C. Mastery without recordings is either decorative or self-rated; neither is worth a column.

### Router shape — `/api/chunks` vs `/api/videos/{id}/chunks`

**Option A:** Nested under `/api/videos/{video_id}/chunks`. Matches 017's autopsy nesting.
- Pros: symmetric with autopsy; per-video listing is the natural shape.
- Cons: Mis frases is a global screen — its primary read is cross-video. A nested router forces a separate "list all" endpoint anyway.

**Option B (chosen):** Top-level `/api/chunks`. POST carries `video_id` in the body.
- Pros: matches the consumer's actual read shape; single router, single resource.
- Cons: drifts from the autopsy nesting precedent.

**Chosen:** B. The reader's mental model wins. `GET /api/chunks` returns the library; the save endpoint takes `video_id` in the body for symmetry.

### `to_response` shape — leaky args vs encapsulated

Decided post-`/simplify`. Initial implementation mirrored `AutopsyRepository.to_response(row, youtube_id)` — args threaded from the route into the repo. Review flagged the route doing ORM-relationship traversal (`row.video.youtube_id`, `row.video.title`) to fill repo args, plus an N+1 on the list path.

**Option A:** Keep mirror — `to_response(row, youtube_id, source_title)`; route digs into `row.video.*` for `GET /api/chunks`.
- Pros: byte-symmetric with 017.
- Cons: leaky abstraction (route knows about ORM relationships); N+1 on list.

**Option B (chosen):** Encapsulate — `to_response(row)` reads `row.video.youtube_id` and `row.video.title` internally; `list_all()` uses `selectinload(Chunk.video)` so the access is free.
- Pros: route layer doesn't traverse the ORM; one extra SELECT total on list instead of N.
- Cons: known drift from 017 — autopsy's `to_response(row, youtube_id)` keeps the leaky shape. Documented in `Spec Divergence`.

**Chosen:** B. 017's pattern is the outlier worth fixing later; chunks gets to be the right shape from day one.

## Tradeoffs

- **Schema independence > storage efficiency.** Standalone chunks means a chunk + its autopsy duplicate the `phrase` field on disk. Trivial cost; the saved cognitive coupling is the win.
- **Persisted prompts > regenerated prompts.** Ana sees the same prompts across sessions, which matches her mental model but means a bad prompt sticks until she deletes the row. There is no edit / PATCH / regen path, intentionally — bad-prompt recovery is "delete and re-save."
- **Top-level `/api/chunks` > nested under `/api/videos`.** Drifts from 017's nesting; aligned to consumer read shape. Per-video listing is out of scope until a consumer asks for it.
- **Encapsulated `to_response(row)` > mirror of 017.** Fixes a latent N+1 and a leaky abstraction; introduces a known drift from autopsy that 017 should eventually align with.
- **No mastery column.** Mis frases ships without mastery in 020 — design's mastery dot becomes a static mock until 021 lands recordings.
- **No backfill, no script.** Unlike 018, there is no `scripts/backfill_chunks.py`. Chunks only exist when Ana saves them; there is no candidate-selection heuristic to backfill against.
- **Two Open Questions parked.** Length cap (200 vs 80) and normalization aggressiveness (strip leading/trailing punctuation?) remain unresolved. Both need real production signal to decide. The status quo (200-char cap, whitespace+casefold-only normalization) is consistent with 017.

### Spec Divergence

The implementation matches the spec acceptance criteria 1-for-1. Three minor implementation-level divergences:

| Spec Said | What Was Built | Reason |
|---|---|---|
| `to_response(row, youtube_id, source_title)` — repo takes scalars from the route | `to_response(row)` — repo reads `row.video.youtube_id` / `row.video.title` internally | Post-`/simplify` review flagged the original shape as a leaky abstraction + latent N+1. Encapsulated version fixes both. Known drift from 017's autopsy `to_response(row, youtube_id)` which stays leaky for now. |
| 502 message: `"Generación de prompts fallida: {exc}"` | 502 message: `"Generación de consignas fallida: {exc}"` | "consignas" is the prompt's own term for these things; using it everywhere keeps the user-facing Spanish consistent. Inconsequential. |

## Spec Gaps Exposed

- **017's `to_response(row, youtube_id)` carries the same leaky-arg / latent-N+1 pattern that 019's `/simplify` pass fixed.** `tests/test_autopsy_routes.py` doesn't trigger the N+1 in practice (its list test inserts ≤2 rows), but the shape is identical. Worth a small follow-up to align autopsy: `AutopsyRepository.to_response(row)` + `selectinload(PhraseAutopsy.video)` in `list_for_video`. Not blocking anything; cleanup item.
- **The Anthropic prompt for chunks lives in `src/services/chunk_prompts.py:_PROMPT_TEMPLATE` and duplicates the `_FENCED_JSON_RE` / `_BARE_*_RE` / `_extract_json` triad now present in three services (`phrase_autopsy.py`, `phrase_markers.py`, `chunk_prompts.py`).** Three copies is the threshold where future drift becomes likely. The 019 plan explicitly deferred extraction; surface here as a candidate for a small refactor *before* a fourth Claude service lands. Most natural shape: `services/_json_extract.py` with `extract_json(text, expect: Literal["array","object","any"])`.
- **`normalize_phrase` lives in `repositories.autopsy_repository` but is now used by `chunks` too.** Cross-feature dependency reads as incidental. A move to a phrase-key module (or to `repositories/__init__.py`) would make the shared use intentional. Low priority.
- **No follow-up for the two Open Questions.** Both depend on real production signal; revisit when 020 is in users' hands.

### Spike (transient, ran 2026-05-14)

7 phrases × 2 runs = 14 Claude calls against `claude-4-sonnet-20250514`, MAX_TOKENS=1200. Script (`scripts/spike_chunk_prompts.py`) deleted post-run, matching 017's pattern.

| metric | result |
|---|---|
| `json.loads` success | 14 / 14 |
| Validator pass (list, 2-4 strs) | 14 / 14 |
| Within-run near-duplicates | 0 / 14 |
| Invitational (manual scan — uses phrase, no analysis/translation) | 28 / 28 |
| Spanish-only, no English crutches | 28 / 28 |
| API failures | 0 |

**Good** — `«no me da igual»`, run 1 (idiomatic, four distinct social scenarios):

```json
[
  "Cuéntame sobre algo que te moleste cuando ves cómo se comportan otras personas. Usa 'no me da igual' para expresar tu opinión.",
  "Imagina que tu amigo dice que todos los políticos son iguales. Respóndele usando 'no me da igual' para mostrar que no estás de acuerdo.",
  "Piensa en una situación donde alguien te dice que algo no importa, pero para ti sí importa. Explica tu posición con 'no me da igual'.",
  "Tu compañero de trabajo llega siempre tarde y dice que no pasa nada. Háblale usando 'no me da igual' para expresar cómo te sientes."
]
```

**Weak** — `«¿sabes?»`, run 2 (all four consignas drift reflective, none social-conversational; schema-safe but a prompt-iteration target if it shows up in production):

```json
[
  "Cuéntale a un amigo algo que realmente te molesta de la sociedad actual, usando «¿sabes?» para enfatizar tu punto.",
  "Describe una situación en la que defendiste tus valores aunque otros no los entendieran, incluyendo «¿sabes?» en tu relato.",
  "Explícale a alguien por qué algo que parece insignificante es muy importante para ti, terminando con «¿sabes?»",
  "Cuenta sobre algo que te apasiona pero que la gente suele considerar aburrido, usando «¿sabes?» para conectar con tu interlocutor."
]
```

**Surprise**: `«pues»` — the case I worried about most (bare filler, minimal semantic content) — produced perfectly serviceable invitational consignas both runs. Concern was misplaced.

The locked production prompt lives in `src/services/chunk_prompts.py:_PROMPT_TEMPLATE` and is byte-identical to the spike's V1.
