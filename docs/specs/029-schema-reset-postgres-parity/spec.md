# Spec: Schema reset — models authoritative + Postgres test parity

| status | approved |

## Why <!-- required -->

`alembic revision --autogenerate` is no longer trustworthy. Every run re-proposes
unrelated FK / index / constraint churn on `answer_progress`, `frase_exercise`,
`processing_jobs`, and even the freshly-added `recordings` table — because the
SQLModel table classes **under-declare** what the migrations actually built (FK
`ondelete` clauses, the `unique_video_question_progress` constraint, some
indexes). On top of that, the test suite runs on in-memory **SQLite** while
dev/prod run **Postgres**, which adds a second layer of dialect drift (`TEXT` vs
`VARCHAR`, a hand-added SQLite "dialect guard" in migration `c6440d9b9453`).

The combined effect: every schema change forces a developer to hand-strip
autogenerated noise, and a *real* change can hide in that noise and be lost. This
already happened twice in the 021 work (the recordings migration was hand-stripped;
the no-store fix's table drift was first seen here).

This feature makes the **models the single source of truth** and the **test engine
match production**, so autogenerate produces an empty diff and tests exercise the
real database. It is pure infrastructure hygiene — no product behavior changes.

### Consumer Impact <!-- required -->

The consumer is the **developer/operator** (not Ana, the end user — this is
invisible to her). The benefit: `alembic revision --autogenerate` becomes
reliable (empty diff when models are unchanged; only intended deltas when they
change), migrations stop needing manual surgery, and the test suite catches
Postgres-specific behavior (types, constraints, sequences, JSON ops) that SQLite
silently papered over. This de-risks every future schema-touching feature,
starting with 022 (user profile + KPIs).

### Roadmap Fit <!-- required -->

Flagged as the open follow-up in the 021 decision record and PR #14. Sequenced
**before 022** (the next feature that adds tables) precisely so 022 is built on a
trustworthy autogenerate baseline rather than fighting the drift. Depends on
nothing; unblocks clean schema work thereafter. Safe to do now because the project
is **local-only with throwaway test data** — no deployed environment is pinned to
the existing migration revision hashes, so history can be rewritten.

## What <!-- required -->

### Acceptance Criteria <!-- required -->

- [ ] After the work, `alembic upgrade head` on a fresh database followed by
  `alembic revision --autogenerate` produces an **empty diff** (no spurious
  table/FK/index/constraint changes). This is the definitive success test.
- [ ] The SQLModel classes declare their full intent: FK `ondelete` on
  `answer_progress`, `frase_exercise`, `processing_jobs`, and `recordings`; the
  `unique_video_question_progress` constraint on `AnswerProgress`; the intended FK
  indexes; `processing_jobs.error` typed as `TEXT`.
- [ ] The `unique_video_question_progress` constraint is **preserved** (the
  `save_answer` upsert in `progress_repository` relies on one row per
  `(video_id, question_id)`).
- [ ] Migration history is collapsed to a **single Postgres-native baseline**
  generated from the corrected models; the 8 prior migration files and the SQLite
  dialect guard are removed.
- [ ] The test suite runs against **Postgres** (not SQLite); all existing tests
  pass unchanged in behavior (114 at time of writing).
- [ ] CI provisions Postgres and the suite is green in CI.
- [ ] `make dev` / local `alembic upgrade head` still bring up a working schema
  from the single baseline.

### Non-Goals <!-- required -->

- **No product behavior or data changes.** Schema *intent* is unchanged; this only
  makes the declarations honest and the engine consistent.
- **No preservation of migration history.** Squashing rewrites it — acceptable only
  because there is no deployed environment depending on the old revision hashes.
- **No new indexes/constraints beyond reconciling declared intent.** Not an
  optimization pass; don't go index-happy.
- **No fix for other OVERVIEW audit items** (eager API-key validation, duplicate
  dialect classifier, stale model IDs) — those are separate.
- **No change to what the schema models** — same tables, columns, relationships.

### Open Questions <!-- optional -->

1. **Test-Postgres provisioning mechanism.** `testcontainers[postgres]`
   (self-contained, spins a container per session — needs Docker locally and in
   CI) vs a plain `DATABASE_URL_TEST` pointing at an externally-running Postgres
   (GH Actions `services: postgres` in CI; local devs use their `make dev`
   Postgres or a compose service). **Leaning:** testcontainers for local+CI
   uniformity, with `DATABASE_URL_TEST` honored as an override/fallback when set.
   Settle during implementation after a quick spike confirms testcontainers
   startup time is acceptable.
2. **Test isolation strategy on Postgres.** Per-test schema create/drop (simple,
   slower) vs create-once-per-session + per-test transaction rollback (faster,
   more fixture machinery). **Leaning:** session-scoped schema (one
   `alembic upgrade head`) + function-scoped transactional rollback, to keep the
   suite fast against a real engine. Confirm during implementation.

### Open Questions resolved before `approved`

Both OQs are implementation-detail leanings, not blockers — explicitly deferred to
implementation with stated defaults. The schema-content decisions (keep the unique
constraint, `error`→TEXT, models authoritative) are settled.

## How <!-- required -->

### Approach <!-- required -->

Three moves, in order:

**1. Make the models authoritative** (`src/models/database.py`). Declare what the
migrations currently hold so model metadata == deployed schema:
- FK `ondelete` via explicit `sa_column=Column(Integer, ForeignKey("…", ondelete=…))`
  (or `Field(sa_column=…)`) on the four drifting FKs — matching the existing
  migration semantics (`CASCADE` where the migrations had it; `SET NULL` on
  `processing_jobs.video_id`).
- Re-add `UniqueConstraint("video_id", "question_id",
  name="unique_video_question_progress")` to `AnswerProgress.__table_args__`.
- Type `processing_jobs.error` as `TEXT` via `sa_column=Column(Text)` (DB is
  already TEXT; this aligns the model down — no column alter needed).
- Keep the FK `index=True` the models already declare; the baseline will create
  those indexes so the declaration is fulfilled.

**2. Squash to one Postgres-native baseline.**
- Delete all files under `alembic/versions/` (the 8 existing migrations).
- With an empty Postgres database and the corrected models, run
  `alembic revision --autogenerate -m "baseline schema"` → a single migration
  that creates every table exactly as the models declare (FKs with ondelete,
  unique constraint, indexes — all present, so no future drift).
- Remove the SQLite dialect guard (no longer needed once tests are Postgres).
- Recreate the local dev DB from the baseline (`drop`/`create` + `upgrade head`).

**3. Switch the test suite to Postgres** (`tests/conftest.py`, `pyproject.toml`,
`.github/workflows/ci.yml`).
- Replace the in-memory SQLite `engine` fixture (currently forces
  `sqlite:///:memory:` and PRAGMA foreign_keys) with a Postgres-backed fixture:
  provision via `testcontainers[postgres]` (added to the `dev` dependency group),
  honoring `DATABASE_URL_TEST` as an override.
- Session-scoped schema via `alembic upgrade head` against the test DB; per-test
  isolation via transactional rollback.
- Drop the SQLite-specific bits (the `PRAGMA foreign_keys = ON` listener — Postgres
  enforces FKs natively; the `StaticPool` / `check_same_thread` args).
- CI: add a `postgres` service (or let testcontainers run under the GH Actions
  Docker daemon) so `uv run pytest` is green in CI.

**Definitive verification** (the falsification test): on a fresh Postgres,
`alembic upgrade head` then `alembic revision --autogenerate` must report **no
changes**. Plus the full suite green on the new fixture.

### Confidence <!-- required -->

**Level:** Medium

**Rationale:** Moves 1–2 (model alignment + squash) are mechanical and have a
crisp, runnable success test (empty autogenerate diff). The uncertainty is move 3:
swapping the test engine touches the fixture design (isolation strategy), adds a
Docker/testcontainers dependency, and needs CI plumbing — more surface than the
schema work itself. SQLModel's `sa_column` FK-with-`ondelete` declaration also has
sharp edges (getting the autogenerate comparison to render *identically* to the
baseline can take an iteration or two).

**Validate before proceeding:**
- Spike the testcontainers Postgres fixture early — confirm container startup time
  is acceptable and the session-schema + transactional-rollback pattern works with
  SQLModel sessions, before converting all test files.
- After the model edits + baseline, run the empty-diff check against Postgres
  *first*; iterate the `sa_column` declarations until autogenerate is silent.

### Key Decisions <!-- optional -->

- **Squash over reconciling patch migration.** No production data to preserve, so a
  single honest baseline beats stacking a corrective migration on accreted history.
  Fewer artifacts, no compat hacks, autogenerate clean by construction.
- **Postgres for tests over keeping SQLite.** Parity with prod removes the
  dialect-drift class entirely and lets the baseline be Postgres-native. Cost: the
  fast zero-dep in-memory suite is replaced by a Docker-backed one — accepted for
  the parity and honesty gains, given the project is committed to Postgres in prod.
- **`error`→TEXT by aligning the model (not altering the DB).** TEXT is the better
  type for error strings and the DB already has it; declaring the model as TEXT
  needs no column alter.
- **Keep the unique constraint.** App-level upsert depends on it; dropping it would
  permit duplicate progress rows.

### Testing Approach <!-- required -->

Per OVERVIEW, the backend has a pytest suite (the 025 infrastructure). This feature
**changes the suite's engine** but not its intent — the existing tests are the
regression guard that schema behavior is unchanged.

- **The empty-diff check is the headline test** for this feature: a script/manual
  step asserting `alembic revision --autogenerate` yields no operations against a
  fresh Postgres at `head`. Consider a small CI step or a documented command.
- **Full existing suite green on Postgres** — all 114 tests must pass against the
  new fixture with no behavioral edits (only fixture/engine changes). Any test that
  was silently relying on SQLite leniency surfaces here and is fixed.
- **FK cascade coverage already exists** (`test_recording_routes.py`'s
  chunk-delete-cascade test, `answer_progress` reset) — these now run on Postgres,
  validating the declared `ondelete` for real rather than via the SQLite PRAGMA.
- **CI**: `uv run pytest` green with Postgres provisioned; lint/format unchanged.
